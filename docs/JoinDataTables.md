# Техническая документация: Соединить две таблицы (JoinDataTables)

Компонент разработан для платформы автоматизации PIX Studio и предназначен для реляционного объединения (Join) двух таблиц данных (`System.Data.DataTable`).

Основная особенность компонента — поддержка независимых списков ключевых полей для левой и правой таблиц, что позволяет связывать структуры с различающимися именами идентификаторов.

## Общие сведения

* **Отображаемое имя в PIX Studio:** Соединить две таблицы
* **Путь в палитре активностей:** `Custom activities`
* **Формула представления (Representation):** `[FirstTable] + [SecondTable] = [ResultTable]`

Активность решает классическую задачу RPA‑сценариев: объединение таблиц из разнородных источников (например, сопоставление выгрузки из CRM или BPMSoft, где ключ называется `new_IDt`, с внутренней базой ORACLE, где этот же ключ называется `ID_CONT`).

| :--- | :--- | :--- | :---: | :--- |

## Режимы объединения (JoinType)

При сопоставлении значений полей системное значение `DBNull.Value` признаётся валидным. Логика сравнения считает, что `NULL == NULL`.

* **Inner**: В итоговую выборку попадают только те строки, у которых композитный ключ левой таблицы полностью совпал с композитным ключом правой таблицы.
* **Left**: В результат гарантированно включаются все строки левой таблицы. Если совпадения в правой таблице нет, то все новые колонки, пришедшие из правой таблицы, заполняются значениями `DBNull.Value`.
* **Right**: В результат гарантированно включаются все строки правой таблицы. Если совпадения в левой таблице нет, левые не‑ключевые колонки заполняются `DBNull.Value`. При этом в левые ключевые колонки принудительно транслируются значения ключа из правой таблицы, чтобы идентификатор строки не был утерян.
* **Full**: Полное внешнее объединение. Включает в себя все строки из обеих таблиц, связывая совпавшие и дополняя `DBNull.Value` недостающие данные с любой из сторон.

## Алгоритм обработки коллизий имён

При формировании результирующей структуры данных (`CreateResultStructure`) действуют строгие правила автоматического разрешения конфликтов во избежание генерации исключения `DuplicateNameException` в ADO.NET:

1. **Исключение дубликатов ключей:** Колонки, перечисленные в `RightJoinKeys`, полностью отсекаются при создании схемы и не попадают в результат под своим именем. Физически их роль начинают выполнять соответствующие колонки из `LeftJoinKeys`.
2. **Разрешение конфликтов не‑ключевых полей:** Если в правой таблице присутствует не‑ключевая колонка, имя которой полностью совпадает с любой колонкой из левой таблицы, алгоритм динамически переименовывает её, добавляя суффикс `_Right`.
3. **Безопасное замещение («Грязная» перезапись):** Если в настройке `SaveDestination` указан режим `LeftTable` или `RightTable`, перед записью данных целевая таблица подвергается жёсткой очистке структуры:

    ```csharp
    target.Constraints.Clear();
    target.Rows.Clear();
    target.Columns.Clear();
    ```

    Это гарантирует, что старая схема данных таблицы‑приёмника будет полностью стёрта, исключая появление смешанных колонок‑мутантов.

## Валидация данных и исключения

Перед запуском основного цикла обработки метод `Execute` выполняет пре‑валидацию бизнес‑логики. Ошибки генерируются в следующих случаях:

* `ArgumentNullException`: Свойство `FirstTable` или `SecondTable` не инициализировано (равно `null`).
* `ArgumentException`:
    * Списки `LeftJoinKeys` или `RightJoinKeys` не заданы или имеют длину 0.
    * «Количество ключей в левой и правой таблицах должно совпадать» — если длины переданных списков не равны.
    * «Левая/Правая таблица не содержит ключ '{key}'» — если строка в списке ключей не соответствует ни одной существующей колонке в таблице.
    * «Тип поля ключа отличается…» — если типы данных сопоставляемых по индексу полей не эквивалентны (например, попытка связать `System.Int32` и `System.String`). Данная проверка выполняется строго по системным типам схемы метаданных и падает даже на пустых таблицах, содержащих 0 строк.

## Архитектура и производительность

Вместо классического вложенного цикла со сложностью $O(N \times M)$, приводящего к зависанию робота на больших объёмах данных, в активности применён подход Hash Join:

* Строки левой и правой таблиц группируются в хэш‑таблицы (`Dictionary<CompositeKey, List<DataRow>>`).
* В качестве ключа словаря выступает разработанный неизменяемый класс `CompositeKey`.
* В классе `CompositeKey` переопределены методы `Equals` и `GetHashCode` на основе вычисления хэша массива объектов (`unchecked` перемножение сдвигов на простые числа 17 и 23). Данный подход обеспечивает корректное сравнение составных ключей любой длины и типов за константное время $O(1)$.
* Результирующая сложность алгоритма составляет $O(N + M)$, что позволяет обрабатывать таблицы на сотни тысяч строк за доли секунды.

## Комплексный пример работы

### Исходное состояние таблиц

**Левая таблица (FirstTable):**

| ID_Client (int) | City (string) |
| --- | --- |
| 1 | Москва |
| 2 | Санкт‑Петербург |

**Правая таблица (SecondTable):**

| CustomerID (int) | City (string) | Manager (string) |
| --- | --- | --- |
| 1 | Казань | Иванов |
| 3 | Новосибирск | Петров |

### Параметры конфигурации в PIX Studio

* `ConnectionType`: `JoinType.Left`
* `LeftJoinKeys`: `["ID_Client"]`
* `RightJoinKeys`: `["CustomerID"]`
* `SaveDestination`: `SaveResultTo.NewTable`

### Результат (ResultTable)

Обратите внимание, что колонка `CustomerID` была удалена. Колонка `City` из правой таблицы получила суффикс `_Right` из‑за конфликта имён с левой таблицей, а для отсутствующего ключа 2 проставились значения `DBNull`.

| ID_Client (int) | City (string) | City_Right (string) | Manager (string) |
| --- | --- | --- | --- |
| 1 | Москва | Казань | Иванов |
| 2 | Санкт‑Петербург | DBNull | DBNull |
